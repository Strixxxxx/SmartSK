Your backend application is running correctly, but Azure's Health Check system cannot get a successful response from it. This is due to a simple path inconsistency between your application's code and your Azure configuration.

Problem Analysis
Your Application is Running: The latest logs from console.txt confirm that your container is starting successfully without crashing. Entries like 

Container ... initialized successfully and is ready to serve requests show that the previous startup errors have been resolved. Your observation that the frontend can use the backend is correct.

API Path Inconsistency: In your main.js file, every functional API route that your frontend uses is prefixed with /api.

app.use('/api/login', loginRouter);

app.use('/api/projects', projectSubmissionRouter);

app.use('/api/admin', adminRouter);

However, your health check endpoint is defined at the root level:

JavaScript

// main.js
app.get('/health', (req, res) => {
  res.status(200).send('OK');
});
The Root Cause: Your Azure Health Check is configured to ping the path /health. While your code defines this route, it's possible that Azure's internal networking or some other configuration is preventing access to non-/api prefixed routes. Since all your working routes use the /api prefix, the most reliable solution is to make the health check consistent with the rest of your application's API structure. The "Degraded" status is likely caused by the health checker receiving a 404 Not Found error or a timeout when it tries to access /health.

How to Fix It
To resolve this, you need to make two small changes: one in your code and one in the Azure portal.

Step 1: Update the Route in main.js
Modify your health check endpoint in main.js to include the /api prefix, just like your other routes.

Change this:

JavaScript

// Health Check Endpoint
app.get('/health', (req, res) => {
  res.status(200).send('OK');
});
To this:

JavaScript

// Health Check Endpoint
app.get('/api/health', (req, res) => {
  res.status(200).send('OK');
});
Step 2: Update the Path in Azure Portal
Now, tell Azure to ping the new, correct path.

Go to your smartSK-backend App Service in the Azure portal.

In the left-hand menu, scroll down to the Monitoring section and click on Health check.

In the Health probe path field, change the value from /health to /api/health.

Click Save at the top of the page.

After you commit and deploy the code change and save the new path in Azure, the health check status on your app's "Overview" page should change from "Degraded" to "Healthy" within a few minutes.

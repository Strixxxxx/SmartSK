ere’s a focused diagnosis of the latest Azure App Service logs you shared, pulling together all the earlier evidence (your code, environment-variable checks, and previous logs):

1. Key Symptoms in the Logs

Repeated container restarts and timeouts

ERROR - Container ... didn't respond to HTTP pings on port: 3000. Failing site start.


Azure’s health probe never gets a successful HTTP response on the expected port.

Conflicting port expectations

The App Service middleware shows:

→ Azure exposes 8181 internally but also injects PORT=3000.

Later:

ERROR - Container ... didn't respond to HTTP pings on port: 3000


→ Azure is checking 3000 for readiness.

Earlier MODULE_NOT_FOUND crash (22 Sept)
That shows at least one historical deploy failed because a required module was missing.
You may have fixed that already, since later logs don’t repeat it.

2. Likely Root Cause

Your app never sends an HTTP response on the port Azure is probing.

The App Service Linux container expects the process to listen on the port provided in the PORT environment variable.

Azure set PORT=3000 (even though it also exposes 8181 internally).

If your Node server starts on a different port or doesn’t bind to 0.0.0.0, the health probe fails and the container is killed.

You previously showed:

const PORT = process.env.PORT || 3000;
app.listen(PORT, () => console.log(`Server is running on port ${PORT}`));


That is correct if process.env.PORT is really 3000 at runtime and the app binds to all interfaces.

But notice this mismatch in the startup command:

docker run ... --expose=8181 ... -e PORT=3000


Azure internally maps 8181 → 3000, so the container is reachable on 8181 from the host network, but Azure’s health probe uses 3000 inside the container.
If your code tries to listen only on 8181 or localhost, it won’t be reachable.

3. Secondary Findings

File-lock warning (Could not find file '/home/site/locks/deployment/info.lock') is a benign Kudu artifact, not the cause of the crash.

The old MODULE_NOT_FOUND was likely from a missing dependency before you rebuilt.

4. Recommended Fixes

Force the server to bind to all interfaces on the injected port
Make sure your app.listen is:

app.listen(process.env.PORT, '0.0.0.0', () => {
  console.log(`Server running on ${process.env.PORT}`);
});


Explicitly add '0.0.0.0' as the host.

Remove any hard-coded PORT in Azure
In the Azure Portal → Configuration → Application settings:
Delete any custom PORT entry so Azure can supply its own value.
If you keep a custom PORT, it must match what the service probes (default is the one it injects).

Redeploy and watch logs
After adjusting, redeploy and check Log Stream. You should see:

Server running on port 3000


and the “didn't respond to HTTP pings” errors should stop.

5. If Issues Persist

Verify nothing in package.json start script overrides the port (e.g., node server.js --port 8181).

Make sure no firewall or middleware (like app.use() with custom host checks) blocks the request.

Bottom Line

The repeated didn't respond to HTTP pings on port: 3000 errors mean the Node process never successfully binds to the port Azure probes.
Ensure the app listens on process.env.PORT and host 0.0.0.0, and remove any conflicting PORT setting in Azure’s configuration.

✦ That is an excellent question for clarification. It's a common feature, but it becomes much more complex with an encrypted database.

  Here is the breakdown:

  You are correct that a user might want to log in with either their username or their email. However, with our new encrypted database, this is not technically feasible in an efficient way.

   * Why Email Works: We can look up a user by their email address because we specifically created the emailHash column. When a user types their email, we can hash it and quickly search the
     database for that hash. This is secure and very fast.

   * Why Username Fails: The username column in the database is now encrypted. There is no corresponding usernameHash column. To find a user by their username, we would have to:
       1. Fetch every single user from your database.
       2. Decrypt the username for every user, one by one.
       3. Compare the decrypted username to the one the user provided.

      This process is extremely slow and inefficient. It would not scale as your user base grows and would make logging in with a username take a very long time.

  The Recommended Solution

  The standard and most secure approach, which my previous (cancelled) change was based on, is to standardize on using the email address as the unique identifier for login and password 
  recovery.

  This means:
   1. The frontend login form should ask for the user's Email Address and Password.
   2. The "Forgot Password" feature will only work by entering an email address.
   3. The username field becomes a display name (like "JohnDoe123") that can be shown in the app, but it is not used for authentication.

  This is how the vast majority of modern applications work, and it allows us to maintain high security and performance.

  Given this technical limitation, shall we proceed with modifying the login and forgot password flows to use the email address as the primary identifier?

 Definitive Implementation Plan (Final Version)

  This document serves as the final blueprint for refactoring the AI reporting system into a resilient, high-performance, asynchronous architecture.

  Phase 1: Azure & Environment Configuration

   * Status: Confirmed.
   * Details: The HA_CONTAINER (for historical CSVs) and JSON_CONTAINER (for report outputs) are created and configured in the backend environment variables.

  Phase 2: Database Migration

   * Action: The following SQL script will be placed in Query.txt for manual execution.
   * Details: This script only adds the new rawDataUpTrack table. The existing rawData, rawDataDetails, and [Raw Data] stored procedure are kept to serve as a synchronized backup data source.

    1     -- Create the new table to specifically track CSV file uploads.
    2     -- Existing rawData tables and procedures will be kept.
    3     CREATE TABLE rawDataUpTrack (
    4         uploadID INT PRIMARY KEY IDENTITY(1,1),
    5         fileName NVARCHAR(255) NOT NULL,
    6         fileUrl NVARCHAR(MAX) NOT NULL,
    7         fileHash VARCHAR(64) NOT NULL,
    8         rowCount INT NOT NULL,
    9         uploadedByUserID INT NOT NULL,
   10         createdAt DATETIME NOT NULL DEFAULT GETDATE(),
   11         CONSTRAINT FK_RawDataUpTrack_User FOREIGN KEY (uploadedByUserID) REFERENCES userInfo(userID)
   12     );
   13 
   14     CREATE INDEX IX_RawDataUpTrack_DateUpload ON rawDataUpTrack(createdAt DESC);

  Phase 3: Backend Implementation (`/backend`)

   1. Reconstruct Data Synchronization Endpoint (`rawData.js`):
       * The POST /api/rawdata/upload route will perform a "dual-write" to keep both data sources synchronized.
       * Logic: Upon receiving a CSV, it will start a database transaction. For each row in the CSV, it will perform an "upsert" (UPDATE or INSERT) into the rawData and rawDataDetails tables. If the
          database update succeeds, it will upload the CSV to the HA_CONTAINER and log the event in rawDataUpTrack. If any step fails, the entire transaction is rolled back.

   2. Create the Python AI Engine (`backend/AI/aiJobs.py`):
       * This self-contained script will execute the entire hourly AI job.
       * Data Sourcing (Primary/Fallback Strategy):
           1. Primary: The script will first attempt to fetch and parse all CSVs from the HA_CONTAINER.
           2. Fallback: Only if the primary source is unavailable, the script will connect to the SQL database and execute the `[Raw Data]` stored procedure. The resulting dataset will be loaded into
               a pandas DataFrame.
       * Processing & Output: The script will use the sourced DataFrame to perform all AI/ML tasks and upload the three final JSON reports to the JSON_CONTAINER.

   3. Implement the Node.js Scheduler (`main.js`):
       * A node-cron job will execute the aiJobs.py script once per hour via a direct spawn command, logging its output for monitoring.

   4. Create Report Retrieval Logic (`backend/AIDataRetrieval/`):
       * This directory will contain dataRetrieve.js (to download JSON reports from Azure) and reports.js (to define the frontend-facing API endpoints).

   5. Cleanup:
       * The obsolete pyBridgeFC.js and pyBridgePA.js files will be deleted.

  Phase 4: Frontend Implementation (`/frontend`)

   1. Modify Admin Data Management UI (`/src/components/Admin/Raw Data/`):
       * The existing rawdata.tsx component will be updated. Its "Update Data" feature will trigger the new "upsert" process.
   2. Update Client-Facing Report Pages:
       * Forecast Page (`/src/components/Client/Forecast/`): Will be refactored to fetch data from the new /api/reports/forecast endpoint.
       * Predictive Analysis Page (`/src/components/Client/PredictiveAnalysis/`): Will be refactored to fetch data from /api/reports/pa-analysis and /api/reports/pa-trends, with all filtering
         handled instantly on the client-side.

  Phase 5: Risk Mitigation & Resilience

   1. Hourly Job Failure:
       * Mitigation: The aiJobs.py script will use an "atomic overwrite" method (saving new reports only on full success) to ensure the old, valid reports remain online if a job fails. The node-cron
          job will have a lightweight retry mechanism.
   2. Corrupt Input Data:
       * Mitigation: A two-layered defense: rawData.js will perform validation on CSVs at upload time, and aiJobs.py will use try...except blocks to abort safely if it encounters a corrupt file,
         preventing a system crash.
